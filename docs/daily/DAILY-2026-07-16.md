## 二、根因分析（指标对不上的原因）

### 2.1 评测口径不同

| 维度 | ShelfPickSense benchmark | 本仓库 `evaluate_inference_upload` |
|------|--------------------------|-------------------------------------|
| 统计单位 | **帧级**二分类（`compute_picking_metrics`） | **段级**事件检测（`evaluate_segments`） |
| 正样本 | `verified_true` 逐帧 `is_picking` | 连续相同货框合并为段 |
| TP/FN | 正帧命中/漏帧 | 段内任一告警即检出 / 整段漏报 |
| FP | 负帧误判为取货 | 每条未覆盖标真段的 `(frame, box)` 告警 |
| Precision | `tp/(tp+fp)` 帧级 | `precision_proxy` = 段 TP/(段 TP+告警 FP) |

### 2.2 ML 推理帧集合不同

- ShelfPickSense 训练与 benchmark：**skeleton.parquet 全帧**推理（Test 5 条共 4892 帧）。
- 本仓库 export：**仅 `pose_frame_interval=2` 的 export 帧**写出结果；若用全 skeleton 帧计分且未导出帧 `pred=False`，取货 Recall 会从 73% 跌到 37%。
- **仅 interval=2 子集计分**时，export ml-only 与原仓库 ML **2445/2445 帧预测完全一致**（`pred_mismatch=0`）。

### 2.3 collision 数据源与管线差异（非两套完全不同算法）

| 项目 | benchmark 侧（`box_human_det`） | 本仓库 baseline export |
|------|--------------------------------|------------------------|
| 数据路径 | `data28/Test/record_XXX/skeleton.parquet` | 记录包 `timeline.parquet` + `skeleton.parquet` 合并 |
| 货框 | `build_box_index_for_record` | `_load_boxes_for_wrist_features`（文件或 manifest 内嵌） |
| 抽帧 | `(frame_idx-1)%2==0` 调 processor | baseline clip JSON 字面帧列表 |
| 引擎 | `box_human_det/collision.py` | `event_engine/collision.py`（同源移植） |

Test 5 帧级：rule_collision TP/FP = **125/25**，本仓库 export = **123/26**（约 98% 一致）。

## 三、实现修改

### 3.1 ML 推理与融合核心

新增 `event_engine/ml_picking_predict.py`：

- 加载 `SklearnPickingModel`、构建 `RecordData`、按帧批量 `predict_frame`

```text

```

## 九、结论摘要

1. **ShelfPickSense benchmark 公式无错误**；本仓库 ML 集成在相同帧上预测一致。
2. **截图 0.89 指标不可直接与段级 FP=93/1403 对比**；上线 KPI 应以 manifest28 段级 eval 为准。
3. **推荐方案**：`rule-baseline` + **ML 否决（A fullfeat）**，非纯 ML。
4. **collision** 为 `box_human_det` 同源移植；差异来自 timeline/skeleton 合并、export 帧列表、annotation 来源，非完全不同的规则。
