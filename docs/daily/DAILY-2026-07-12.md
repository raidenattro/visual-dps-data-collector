## 补充：回放特征加载拖慢视频（2026-07-12）

### 一、问题背景

| 项 | 内容 |
|----|------|
| 模块 | 回放打开记录、视频首屏加载 |
| 问题 | 新增骨骼特征后，打开记录加载视频明显变慢 |
| 影响 | 视频/骨架与特征 API 争抢后端，首屏卡顿 |
| 环境 | Web SPA + FastAPI 同步端点 |

### 二、根因分析

| 类型 | 原因 |
|----|------|
| 后端 | `/playback-features` 每请求读 Parquet 多帧并重算速度/角度，默认还读手腕 Parquet |
| 前端 | `renderFrameEntry` 每帧调用 `updatePlaybackSkeletonFeaturesUi`；特征加载完成又 `redrawCurrentFrame()` 形成请求+重绘风暴 |
| 架构 | 打开记录时 `redrawCurrentFrame` 与视频加载并行，首屏即触发重型 API |

### 三、实现修改

修改文件：

- `api/playback_features_service.py`：进程内 LRU 缓存 `(record_id, frame_idx, include_wrist)`；默认 `include_wrist=false` 跳过手腕 Parquet
- `api/routes/http.py`：新增查询参数 `include_wrist`
- `web/app/13-playback-skeleton-features.js`：视频就绪前不请求；播放中仅用缓存；暂停 debounce 280ms；移除 `redrawCurrentFrame` 循环
- `web/app/10-render-collision.js`：播放中跳过特征 UI 更新；新增 `repaintFeatureTrackLabelsOnly` 轻量刷新标签
- `web/app/06-records.js`：视频加载完成后延迟 700ms 再 `enablePlaybackSkeletonFeatureFetch`

### 四、改动文件索引

| 文件路径 | 修改说明 |
|----|----|
| api/playback_features_service.py | LRU 缓存 + 可选手腕 |
| api/routes/http.py | include_wrist 参数 |
| web/app/13-playback-skeleton-features.js | 延迟/防抖/播放守卫 |
| web/app/10-render-collision.js | 播放跳过 API + 轻量标签重绘 |
| web/app/06-records.js | 视频就绪后启用特征拉取 |

|----|------|

## 补充：回放特征与 export 路径/帧号对齐（2026-07-12 夜）

### 一、问题背景

| 项 | 内容 |
|----|------|
| 模块 | 回放特征 API / 时间轴帧索引 / 准确率跳转 |
| 问题 | 回放用 lookback 全帧或合成 1..N 密集索引，与 export 抽帧（`pose_frame_interval=2`）及 upload clip `frame_idx` 不一致 |
| 影响 | 暂停查看的 `ankle_max_speed`、门控 block 与准确率诊断/export 脚本结果不可对照 |
| 环境 | main 分支，config `pose_frame_interval=1`，实验 baseline 为 2 |

### 二、根因分析

| 类型 | 原因 |
|----|------|
| 后端 | `playback_features_service` 在 lookback 窗口内对全部帧算速度/角度，未走 `load_all_frames` + export_indices 路径 |
| 前端 | `buildSyntheticFrameIndexFromManifest` / `densifyFrameIndexFromManifest` 将时间轴补齐为 1..N，破坏 export 帧号 |
| 数据 | Parquet 本地 dense 时 `frame_idx==source_frame_idx`，但特征语义应以 export 抽帧集合为准 |

### 三、实现修改

### 回放特征走 export 同路径

修改文件:

- `event_engine/export_frame_utils.py`（新增）
- `event_engine/skeleton_features.py`
- `api/playback_features_service.py`

修改内容:

- 共享 `export_frame_key`、`resolve_export_indices`（优先 baseline clip，否则 timeline + interval=2）
- `filter_frames_to_indices` 按 `source_frame_idx` 匹配，输出 `frame_idx` 统一为 export 键
- 回放 API：`load_all_frames` → 全 timeline 上 `extract_subsampled_velocity/angle` → 按请求 `frame_idx` 切片；非 export 帧返回 `is_export_frame: false`
- 按 record 缓存 merged 特征，避免每帧重复全量计算

修改原因:

- 与 `upload_export_common` / 准确率评估同一抽帧集合与帧号

影响:

- 暂停在 export 帧（如 FN 帧 91）时速度与门控可与 export 脚本对照

### 前端帧号对齐

修改文件:

- `web/app/10-render-collision.js`
- `web/app/13-playback-skeleton-features.js`
- `web/index.html`

修改内容:

- timeline / v1 frames 索引优先 `source_frame_idx`
- 移除合成密集 1..N 与 densify 逻辑
- 侧栏展示 `is_export_frame`、`pose_frame_interval`、`export_indices_source`；非 export 帧提示「仅作参考」

### 四、改动文件索引

| 文件路径 | 修改说明 |
|----|----|
| event_engine/export_frame_utils.py | export 抽帧索引解析（新增） |
