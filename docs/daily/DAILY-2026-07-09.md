## 一、问题背景

| 项 | 内容 |
|----|------|
| 模块 | 准确率评估页（`web/accuracy.js`）、上传推测评估 API（`api/inference_eval_service.py`）、离线评估脚本 |
| 问题 A | 刷新页面或切换任务后，刚完成的 eval run 无法快速找回，需重新上传目录或重跑评估 |
| 问题 B | 上传推测评估目录若含 `accuracy_report.json`、`_accuracy_eval.json` 等报告/索引文件，会被当作 clip JSON 解析导致报错或统计异常 |
| 问题 C | 规则变体（如 speed-lower60）与 baseline 的误报/漏报差异缺少结构化对比工具，人工 diff 成本高 |
| 问题 D | speed-lower60 碰撞速度阈值变体相对 baseline 的 FP/FN  trade-off 未量化，无法评审是否上线 |
| 影响 | 评估迭代效率低；export 目录结构不规范时上传失败；规则调参无法量化「消除了哪些误报、新增了哪些漏报」 |
| 环境 | Windows 10 · Python 3 · 本地 `localdata/export` 与 `localdata/eval_runs` |

## 二、根因分析

| 类型 | 原因 |
|------|------|
| 前端 | 评估结果仅保存在内存变量，未持久化 `eval_id`；无历史 run 下拉与加载入口 |
| 后端 | `discover_upload_json_files` / zip 解压枚举对所有 `*.json` 一视同仁，未排除报告类文件 |
| 数据 | clip 推测 JSON 根节点为帧数组，报告类 JSON 根节点为对象，格式混扫导致解析失败 |
| 算法/评估 | 规则变体对比需按 `(record_id, frame_idx, box_token)` 与 `(record_id, frame_start, frame_end, gt_tokens)` 做集合差分，此前无专用脚本 |

## 三、实现修改

### 1. 准确率页保留历史评估 run

修改文件:

- `web/accuracy.js`
- `web/index.html`
- `web/style.css`
- `config.json`（同提交内附带 `video.transcode_height: 480 → 0`）

修改内容:

- 新增 `#accuracy-eval-history` 下拉、加载/刷新按钮；调用 eval run 列表 API 填充历史条目。
- `rememberEvalResult` 将最新 `eval_id` 写入 `localStorage`（`ACCURACY_LAST_EVAL_KEY`），页面初始化时自动恢复选中项。
- `loadHistoricalEvalRun` 拉取 manifest + summary + clips，复用现有诊断/回放跳转逻辑；结果区标注「历史评估」来源。
- `applyHistoricalEvalParams` 按 manifest 还原评估模式、机位、标签筛选等表单状态。

修改原因:

- 评估 run 已落盘至 `localdata/eval_runs/`，前端需与之对齐，避免重复上传。

影响:

- 用户可在准确率页直接切换历史 run，继续漏报/误报诊断与回放跳转。

### 2. 上传推测评估跳过报告类 JSON

修改文件:

- `api/inference_eval_service.py`
- `api/routes/http.py`
- `web/accuracy.js`
- `web/index.html`

修改内容:

- 新增 `SKIP_INFERENCE_UPLOAD_JSON_NAMES` 与 `is_skipped_inference_upload_json()`，过滤 `_accuracy_eval.json`、`accuracy_report.json`、`collision_variants_build_summary.json` 及 `accuracy_report*` 前缀文件。
- 新增 `is_inference_clip_json_file()`：仅根节点为数组的 JSON 视为可评估 clip。
- zip 上传与目录扫描路径统一走上述过滤；前端上传提示补充「已跳过非 clip 文件」说明。

修改原因:

- export 目录常同时包含 clip 推测结果与评估报告，混扫导致非 clip 文件进入评估流水线。

影响:

- 可直接上传完整 export 目录，无需手动剔除报告 JSON。

### 3. export 误报/漏报对比脚本

修改文件:

- `scripts/data/compare_export_false_alarms.py`（新增）
- `scripts/data/strike_eliminated_fp_in_md.py`（新增）

修改内容:

- `compare_export_false_alarms.py`：读取两目录 `accuracy_report.json`，按帧级误报键与段级漏报键做集合差分，输出 JSON + Markdown 对比报告。
- `strike_eliminated_fp_in_md.py`：在 baseline 误报明细 MD 中，对被实验消除的误报序号加删除线，便于人工复核。

修改原因:

- 规则调参需量化 FP/FN 变化及逐条明细，支撑 speed-lower60 等变体评估。

影响:

- 可一键生成 `compare_{experiment}_vs_{baseline}.md/json`，支撑离线评审。

### 4. rule-speed-lower60 vs baseline 误报/漏报对比评估

工作内容:

- 对 **`localdata/export/rule-baseline-prod-test`**（baseline）与 **`localdata/export/rule-speed-lower60-prod-test`**（实验，碰撞速度阈值 lower60 变体）分别跑上传推测评估，生成各目录 `accuracy_report.json` / `accuracy_report.md` 及 eval run 落盘。

  - Markdown：`localdata/export/compare_rule-speed-lower60-prod-test_vs_rule-baseline-prod-test.md`
  - JSON：`localdata/export/compare_rule-speed-lower60-prod-test_vs_rule-baseline-prod-test.json`

| 对比报告 | — | 2026-07-09 05:16 |

系统级汇总（28 片 RTMPose-M prod-test 样本）:

关键结论:

- speed-lower60 **纯减误报、无新增误报**：77 条 baseline 误报被消除，实验侧未引入任何新误报帧。
